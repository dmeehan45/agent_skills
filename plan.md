# Plan: Create a reusable Codex skill for UX + visual quality reviews

## Executive overview
Build a new Codex skill that standardizes frequent product reviews for:
- accessibility UX standards,
- user-flow simplicity and visual continuity (no dead ends),
- visual cleanliness and cognitive-load reduction,
- mobile responsiveness and workflow completion parity,
- visual design and UX best-practice compliance.

Primary audience: agents (and reviewers) who need consistent, repeatable front-end quality assessments.

Proof of success:
1. Skill validates and packages successfully in Codex skill format.
2. Running the skill on a target UI yields a structured review output with findings, severity, evidence, and actionable fixes.
3. Every requested review area is covered by explicit checklist items and scoring criteria.

## Contracts first (shared interfaces for parallel work)

### Skill identity contract
- **Skill name (folder + frontmatter):** `ux-quality-review`
- **Root path:** `skills/ux-quality-review/`
- **Required file:** `skills/ux-quality-review/SKILL.md`

### Output contract (what the skill must produce)
All review outputs generated by this skill must follow this exact section order:
1. `Scope & Context`
2. `Findings by Category`
   - Accessibility UX standards
   - User flow simplicity & continuity
   - Visual cleanliness & cognitive load
   - Mobile responsiveness & interaction ergonomics
   - UX / visual best-practice compliance
3. `Severity Summary` (Critical / High / Medium / Low counts)
4. `Priority Fix Plan (Now / Next / Later)`
5. `Acceptance Checks`

### Finding-item contract
Each finding must include fields in this sequence:
- `ID`
- `Category`
- `Severity`
- `Evidence`
- `User Impact`
- `Recommendation`
- `Implementation Hint`

### Severity contract
- **Critical:** Blocks task completion, causes major accessibility barriers, or breaks core workflows.
- **High:** Serious friction, likely abandonment/confusion, or strong standards violations.
- **Medium:** Noticeable UX quality degradation but task still completable.
- **Low:** Polish or consistency issue with minor impact.

### Scoring contract
- 0–5 score per category with anchors documented in references.
- Overall status mapping:
  - `0.0–1.9` = Failing
  - `2.0–2.9` = Needs major improvement
  - `3.0–3.9` = Acceptable with gaps
  - `4.0–5.0` = Strong

### Stub file map (exact paths)
- `skills/ux-quality-review/SKILL.md`
- `skills/ux-quality-review/references/review-checklists.md`
- `skills/ux-quality-review/references/scoring-rubric.md`
- `skills/ux-quality-review/references/remediation-patterns.md`
- `skills/ux-quality-review/assets/review-output-template.md`
- `skills/ux-quality-review/assets/agent-brief-template.md`

## Parallel workstreams

## Workstream 1 — Skill shell & trigger design
**Agent role:** Skill Architect

**File ownership:**
- `skills/ux-quality-review/SKILL.md`

**Inputs:**
- Contracts in this plan only.

**Outputs:**
- Completed SKILL.md frontmatter + concise workflow instructions + references navigation.

**Step-by-step tasks:**
1. Initialize skill folder and base SKILL.md using Codex skill conventions.
2. Write precise trigger-rich `description` including all five review domains.
3. Define lightweight execution workflow (intake → assess → report → prioritize).
4. Link to reference files for detailed checklists/rubrics.
5. Ensure SKILL.md remains concise (core guidance only).

**Success criteria (binary):**
- SKILL.md has valid frontmatter with only `name` and `description`.
- Description clearly indicates when this skill should trigger.
- Body points to all reference files in this plan.

**Validation steps:**
- `python scripts/validate_skill.py skills/ux-quality-review` (or packaging validator if standalone validator unavailable) returns success.
- Manual check: SKILL.md body < 500 lines.

**Edge cases + negative tests:**
- Missing category mention in description should fail review.
- Invalid frontmatter keys should fail validation.

## Workstream 2 — Accessibility + flow checklist references
**Agent role:** UX Heuristics Author

**File ownership:**
- `skills/ux-quality-review/references/review-checklists.md`

**Inputs:**
- Contracts in this plan.

**Outputs:**
- Actionable checklist covering accessibility standards + flow continuity requirements.

**Step-by-step tasks:**
1. Create checklist sections for each required review category.
2. Add explicit dead-end detection checks (“what do I do next?” moments).
3. Add mobile-specific completion checks per critical workflow.
4. Include observable pass/fail indicators per checklist item.

**Success criteria (binary):**
- All five requested review domains have dedicated subsections.
- Each subsection includes at least 8 concrete checks.
- Checklist includes at least 5 explicit “dead-end/no-next-step” probes.

**Validation steps:**
- `rg -n "^## |dead-end|next step|mobile|accessibility" skills/ux-quality-review/references/review-checklists.md`
- Manual spot-check that each item is observable (no vague wording).

**Edge cases + negative tests:**
- A check requiring hidden implementation details is invalid.
- A check without observable evidence criteria is invalid.

## Workstream 4 — Remediation patterns library
**Agent role:** UX Improvement Playbook Author

**File ownership:**
- `skills/ux-quality-review/references/remediation-patterns.md`

**Inputs:**
- Findings format contract + category scope.

**Outputs:**
- “Issue pattern → recommended fix” library for rapid actionable recommendations.

**Step-by-step tasks:**
1. List recurring anti-patterns (contrast misuse, excessive typography variants, text-link primary actions, etc.).
2. Provide remediation steps optimized for reduced cognitive load.
3. Include mobile-first interaction guidance (button targets, tap zones, hierarchy).
4. Add “do/don’t” micro-examples for clarity.

**Success criteria (binary):**
- Contains at least 20 issue patterns with paired fixes.
- Includes brand-consistency guidance for color/type/highlight usage.
- Includes explicit mobile parity remediation guidance.

**Validation steps:**
- `rg -n "pattern|fix|Do|Don't|mobile|cognitive|brand" skills/ux-quality-review/references/remediation-patterns.md`
- Manual check that every pattern has concrete action steps.

**Edge cases + negative tests:**
- Recommendations that are purely aesthetic without user-impact rationale are invalid.
- Fixes that require undefined brand rules must include fallback guidance.

## Integration plan (dependency-free)
- Integration strategy: contract-first with isolated file ownership; each workstream edits unique files only.
- No blocking dependencies: each agent can start from `plan.md` contracts immediately.
- Merge order (non-blocking): WS2/WS3/WS4/WS5 can merge in any order; WS1 last to ensure SKILL.md links final filenames.
- Conflict prevention:
  - Single owner per file path.
  - No shared edits to same markdown file.
  - Shared terms locked by contracts above.
- Adapter/stub approach:
  - If one reference file is not ready, SKILL.md can temporarily link placeholder headings by exact filename without changing contract fields.

## Risks and mitigations (parallel-specific)
1. **Contract drift across files**
   - Mitigation: lock required headings/fields in this plan; run `rg` validation checks before merge.
2. **Hidden coupling via terminology mismatch**
   - Mitigation: reuse exact category labels from contract; forbid synonyms in required headings.
3. **Uneven specificity across workstreams**
   - Mitigation: enforce binary success criteria + minimum item counts per workstream.
4. **Template and rubric misalignment**
   - Mitigation: acceptance checklist includes cross-file consistency check between template fields and rubric categories.
5. **Overly verbose SKILL.md causing context bloat**
   - Mitigation: keep SKILL.md minimal; move depth into `references/` and `assets/` files.
